# 세노이드 폐쇄몰 기반 브랜드페이지 구매경로 운영설계

## 1. 설계 배경

오너는 네트워크마케팅 기반 회사이므로 일반 온라인 쇼핑몰처럼 외부 고객이 바로 공개몰에서 결제하는 구조가 아닐 수 있다.  
회원 중심 폐쇄몰 구조에서는 외부 고객에게 결제 링크를 바로 노출하기 어렵고, 영업자별 실적 반영과 회사 주문 접수 흐름도 함께 고려해야 한다.

따라서 외부 고객에게는 결제 페이지보다 먼저 **신뢰 확인용 브랜드페이지**가 필요하다.

또한 세노이드 제품은 관계 연구소/기술 기반 제품으로 설명할 수 있고, 결제 역시 오너 생태계 안의 관계 PG사를 활용할 가능성이 있다.  
따라서 이 설계는 단순한 SNS 랜딩페이지가 아니라 **연구소 신뢰, 제품 브랜드, 영업자 실적, 관계 PG 결제**를 연결하는 브랜드 커머스 PoC로 확장할 수 있다.

### 핵심 방향

> 폐쇄몰은 내부 주문/회원 관리용으로 유지하고,  
> 외부 고객에게는 브랜드페이지를 신뢰 랜딩페이지로 보여준다.

브랜드페이지의 목적은 즉시 결제가 아니라, 고객이 제품과 회사를 믿고 개인 카톡 상담 또는 회사 공식 접수 흐름으로 이동하도록 돕는 것이다.

### 브랜드 커머스 PoC 방향

```text
관계 연구소/기술 신뢰
-> 세노이드 브랜드페이지
-> 오너 영업조직의 개인 카톡 상담
-> 영업담당 버튼으로 개인 주문/결제 링크 발행
-> 관계 PG사 결제창 호출
-> 회사 주문/배송/실적 관리
```

이 방향의 장점은 고객에게는 회사 공식 브랜드 경험을 제공하고, 내부적으로는 영업자별 실적과 결제 데이터를 자동으로 묶을 수 있다는 점이다.

## 2. 전체 구매 경로

```text
인스타그램 발견
-> 브랜드페이지에서 제품/회사 신뢰 확인
-> 네이버 블로그에서 상세 후기/FAQ 확인
-> 개인 카톡 상담
-> 브랜드페이지 하단 영업담당 버튼에서 개인 주문/결제 링크 생성
-> 고객이 링크에서 주문정보 입력
-> 관계 PG사 결제창에서 결제
-> 회사 배송
-> 공식 카카오채널 추가
-> 챗봇/공지/재구매 관리
```

### 단계별 역할

| 단계 | 채널 | 목적 |
|---|---|---|
| 발견 | 인스타그램, 릴스, 스토리 | 제품 호기심 유발 |
| 신뢰 확인 1 | 브랜드페이지 | 회사/제품/구매 절차에 대한 공식 신뢰 제공 |
| 신뢰 확인 2 | 네이버 블로그 | 사용감, FAQ, 기술 설명, 검색 신뢰 보강 |
| 상담 | 개인 카톡 | 관계 기반 고객 고민 확인과 구매 결정 유도 |
| 링크 발행 | 브랜드페이지 영업담당 버튼 | 사번/담당자 코드 기반 개인 주문 링크 생성 |
| 접수/결제 | 브랜드페이지 주문폼 + 관계 PG 결제창 | 회사가 주문, 결제, 배송을 통제 |
| 실적 | 담당자 코드/영업자명 | 영업자 실적 반영 |
| 고객관리 | 공식 카카오채널 + 챗봇 | 사용법, FAQ, 이벤트, 재구매 관리 |

## 3. 브랜드페이지의 역할

브랜드페이지는 결제 중심 페이지가 아니라 **신뢰 중심 랜딩페이지**다.

### 고객이 브랜드페이지에서 확인해야 할 것

- 이 제품이 어떤 제품인지
- 회사가 공식적으로 안내하는 정보인지
- 제품 3종의 차이가 무엇인지
- 어떤 피부 고민에 추천할 수 있는지
- 사용 순서와 주의사항은 무엇인지
- 과장광고가 아니라 안전한 표현으로 설명되어 있는지
- 구매 또는 상담은 어떻게 진행되는지
- 입금 후 어떤 절차로 배송되는지
- 공식 카카오채널에서 어떤 도움을 받을 수 있는지

### 브랜드페이지 필수 섹션

| 섹션 | 내용 |
|---|---|
| Hero | 리버스12 세노이드 3종 대표 이미지, 한 문장 가치 제안 |
| 제품 3종 소개 | 미스트, 에센스, 크림의 역할과 사용 장면 |
| 핵심 기술 | 세놀리틱, 네오스피큘, 다이아 스피큘을 쉬운 말로 설명 |
| 사용 루틴 | 아침/저녁 또는 제품별 사용 순서 |
| FAQ | 민감한 피부, 사용 순서, 사용 주기, 끈적임, 가격 관련 질문 |
| 과장 표현 주의 | 치료/재생/제거가 아니라 관리/케어/도움 중심 안내 |
| 구매/상담 절차 | 개인 카톡 상담, 입금, 주문 접수, 배송 흐름 |
| 담당자 코드 안내 | 실적 반영을 위한 담당자명/코드 입력 안내 |
| 공식 카카오채널 | 구매 후 사용법/FAQ/재구매 안내 채널 추가 유도 |
| 회사 정보 | 공식 회사명, 고객센터, 배송/교환/환불 안내 |

## 4. 결제와 주문 접수 구조

### 일반 상담판매 표준 프로세스

카톡 상담 판매에서 일반적으로 쓰이는 흐름은 **상담은 카톡에서 하고, 결제는 회사 공식 결제 수단으로 분리하는 방식**이다.  
고객은 카톡에서 제품 설명과 구매 결정을 하고, 실제 결제는 회사 계좌, 회사 결제 링크, 톡스토어/자사몰/폐쇄몰 링크 중 하나로 진행한다.

```text
고객 문의
-> 영업사원 개인 카톡 상담
-> 제품/수량/배송정보/담당자 코드 확인
-> 결제 방식 선택
   1. 회사 공식 계좌 입금
   2. 영업사원이 버튼으로 생성한 회사 명의 1회성 카드결제 링크
   3. 회사 공식몰/톡스토어/폐쇄몰 주문 링크
-> 회사가 결제 확인
-> 회사가 주문/배송 처리
-> 영업사원 실적 반영
-> 구매 후 공식 카카오채널 추가
```

핵심은 영업사원이 고객과 관계를 만들고 주문 의사를 확인하되, **돈은 회사 공식 창구로 들어가게 하는 것**이다.  
이렇게 해야 고객 신뢰, 환불/교환, 세무 처리, 배송 처리, 영업자 실적 반영이 한곳에서 관리된다.

### 결제 방식별 현장 사용 구조

| 방식 | 현장 사용 흐름 | 적합한 경우 | 주의 |
|---|---|---|---|
| 회사 계좌 입금 | 상담 후 회사 계좌와 주문 접수 양식 안내 | 가장 빠른 초기 운영 | 입금자명/담당자 코드 누락 방지 필요 |
| 회사 결제 링크 | 영업사원이 주문 정보를 입력하면 회사 명의 1회성 링크가 자동 생성됨 | 카드결제, 간편결제, 고객 신뢰가 중요한 경우 | 회사 명의 PG/링크결제 계약 필요 |
| 공식몰/폐쇄몰 주문 링크 | 고객이 지정 링크에서 직접 주문/결제 | 회원 구조와 주문 시스템이 이미 있는 경우 | 외부 고객 접근 가능 여부 확인 필요 |
| 카카오톡 스토어/톡체크아웃 | 톡스토어 상품 링크로 주문/결제/알림톡 처리 | 공개 판매가 가능한 일반 상품 | 네트워크마케팅 폐쇄몰 정책과 충돌 가능 |
| 페이앱/블로그페이류 링크결제 | 회사 계정에서 결제 링크를 만들어 카톡/SMS로 발송 | 빠른 테스트와 상담판매 | 개인 계정보다 회사 계정 사용 권장 |

오너의 경우에는 폐쇄몰/회원 중심 구조가 있으므로, 초기 운영안과 확장 운영안을 나누어 설계한다.

#### 초기 운영안

초기에는 본사 또는 관리자 1명이 주문 접수를 확인하고 링크를 발행할 수 있다.  
다만 이 방식은 판매량이 늘어나면 병목이 되므로 임시 운영안으로만 둔다.

```text
영업사원 개인 카톡 상담
-> 주문 정보와 담당자 코드 확보
-> 회사에 주문/결제 요청 접수
-> 회사가 계좌 입금 확인 또는 카드결제 링크 발행
-> 회사 배송
-> 영업사원 실적 반영
```

#### 확장 운영안

실제 현장 운영은 영업사원이 고객 앞에서 바로 처리할 수 있어야 한다.

```text
영업사원 개인 카톡 상담
-> 영업자용 주문/결제 화면 접속
-> 제품, 수량, 담당자 코드 입력
-> [결제 링크 만들기] 버튼 클릭
-> 회사 명의 결제 링크 자동 생성
-> [카톡에 붙여넣기] 또는 [공유하기] 버튼으로 고객에게 전달
-> 고객이 링크에서 개인정보/배송정보 입력
-> 고객이 카드결제 또는 계좌이체 선택
-> 회사가 결제/주문/배송을 처리
-> 담당자 코드로 영업자 실적 자동 반영
```

이 구조에서는 회사 직원이 옆에 앉아 링크를 보내는 것이 아니라, **회사 시스템이 링크를 만들고 영업사원은 링크를 전달만 하는 구조**가 된다.

### 영업자 버튼형 링크 발행 설계

영업자가 현장에서 사용할 화면은 복잡하면 안 된다.  
최소 버튼 구조는 다음과 같다.

```text
제품 선택
수량 선택
담당자 코드 자동 입력
고객명 또는 메모 입력
[계좌이체 안내문 만들기]
[카드결제 링크 만들기]
[카톡에 붙여넣기]
```

### 고객 입력형 주문/결제 링크 설계

영업자가 고객 정보를 대신 입력하는 구조보다 더 좋은 방식은, 영업자가 링크만 보내고 고객이 직접 정보를 입력하는 구조다.

```text
영업사원
-> 고객에게 주문/결제 링크 전달

고객
-> 링크 접속
-> 제품/수량 확인
-> 이름/연락처/배송주소 입력
-> 개인정보 수집 동의
-> 결제 방식 선택
-> 카드결제 또는 계좌이체 안내 확인
```

이 방식의 장점은 다음과 같다.

| 장점 | 설명 |
|---|---|
| 개인정보 부담 감소 | 영업사원이 고객 주소/연락처를 직접 받아 보관하지 않아도 됨 |
| 주문 누락 감소 | 고객이 직접 입력하므로 배송정보 누락을 줄일 수 있음 |
| 실적 자동 연결 | 링크에 담당자 코드가 들어가면 실적 반영이 쉬움 |
| 카톡 상담 유지 | 영업사원은 상담과 신뢰 형성에 집중할 수 있음 |
| 회사 통제 강화 | 결제/주문/배송 데이터가 회사 시스템에 모임 |

### 영업사원 지문과 실적 자동 귀속 설계

이 구조는 실제 구현 가능하다.  
다만 URL에 `seller=홍길동`처럼 단순 문자열만 붙이는 방식은 부족하다. 고객이 링크를 복사하거나 누군가 파라미터를 바꾸면 실적 분쟁이 생길 수 있기 때문이다.

실적 귀속은 **서버가 발급한 링크 토큰**을 기준으로 처리한다.

```text
영업사원 로그인
-> 영업자 ID와 담당자 코드 확인
-> 서버가 주문/결제 링크 레코드 생성
-> 링크에 공개 토큰 포함
-> 고객이 링크 접속
-> 서버가 토큰으로 영업사원/제품/금액을 조회
-> 고객이 개인정보/배송정보 입력
-> 결제 완료
-> 결제 웹훅 또는 주문 조회로 결제 완료 확인
-> 주문 레코드에 담당자 ID 자동 귀속
-> 실적/정산/배송 리스트에 자동 반영
```

#### 링크에 묻어야 할 지문

| 값 | 설명 | 노출 여부 |
|---|---|---|
| `salespersonId` | 내부 영업사원 고유 ID | 비노출 |
| `salespersonCode` | 담당자 코드 | 일부 노출 가능 |
| `linkToken` | 서버가 발급한 랜덤 공개 토큰 | 노출 |
| `orderDraftId` | 결제 전 임시 주문 ID | 비노출 또는 토큰화 |
| `campaignId` | 인스타/블로그/카톡 캠페인 구분 | 선택 |
| `productId` | 주문 상품 | 비노출 또는 서버 조회 |
| `amount` | 결제 금액 | 서버 확정 |
| `createdAt/expiresAt` | 링크 생성/만료 시간 | 서버 관리 |

고객에게 보이는 링크는 다음처럼 단순해야 한다.

```text
https://senoid.kr/order/LK_8x2pQ9
```

서버 내부에는 다음처럼 저장한다.

```text
linkToken: LK_8x2pQ9
salespersonId: sp_1024
salespersonCode: OW-CHOI-017
productId: senoid-cream
amount: 128000
status: opened | paid | expired | cancelled
```

#### 변조 방지 원칙

- 고객이 URL의 담당자 코드를 바꿔도 실적이 바뀌면 안 된다.
- 실적 귀속은 URL 문자열이 아니라 서버의 `linkToken` 레코드로 결정한다.
- 링크 생성 시점의 `salespersonId`, `productId`, `amount`를 서버에 저장한다.
- 결제 완료 후에는 담당자 귀속값을 잠근다.
- 수정이 필요하면 관리자 승인 로그를 남긴다.
- 같은 고객이 링크를 다른 사람에게 공유해 결제해도 원 링크 영업사원의 실적으로 본다.

#### 구현 방식 선택지

| 방식 | 구현 난이도 | 실적 자동 귀속 | 권장도 |
|---|---:|---:|---:|
| 자체 주문 랜딩페이지 + PG 결제 | 중간 | 높음 | 가장 권장 |
| PG 링크결제 + 커스텀 파라미터/메모필드 | 중간 | 중간~높음 | PG 지원 여부 확인 |
| 정적 상품 링크 + 담당자 코드 파라미터 | 낮음 | 낮음 | 임시 운영만 |
| Google Form + 계좌이체 | 낮음 | 중간 | 초기 테스트 |

가장 좋은 구조는 **자체 주문 랜딩페이지를 먼저 열고, 그 안에서 결제 PG로 넘기는 방식**이다.  
이렇게 하면 개인정보 동의, 배송정보, 담당자 코드, 주문 상태, 결제 상태를 모두 회사 시스템에서 통제할 수 있다.

### 링크 발송 방식

개인 카카오톡을 시스템이 임의로 자동 발송하는 구조는 플랫폼 정책상 제약이 있을 수 있다.  
따라서 초기에는 자동 발송보다 **링크 생성 후 복사/공유**가 현실적이다.

| 방식 | 설명 | 권장도 |
|---|---|---|
| 링크 복사 | 영업자가 버튼을 누르면 카톡에 붙여넣을 문구가 복사됨 | 높음 |
| 모바일 공유 시트 | 휴대폰 공유 기능으로 카카오톡에 전달 | 높음 |
| 알림톡 발송 | 회사 공식 카카오채널/알림톡 계약을 통해 고객에게 발송 | 중기 목표 |
| 친구톡 발송 | 채널 친구 대상으로 마케팅성 메시지 발송 | 중기 목표 |
| 개인 카톡 자동 발송 | 개인 카톡 계정을 시스템이 조작 | 피해야 함 |

결론적으로 1차 PoC는 **카톡 자동 발송**이 아니라 **카톡에 붙여넣을 링크와 문구를 자동 생성**하는 수준으로 시작한다.

PG사가 이미 있다면 새 PG를 찾기보다 먼저 아래 항목을 확인한다.

```text
1. 주문별 1회성 결제 링크를 만들 수 있는가?
2. 결제 링크에 주문번호/담당자 코드/메모를 넣을 수 있는가?
3. 결제 완료 내역을 엑셀 또는 API로 받을 수 있는가?
4. 부분취소/환불 처리가 가능한가?
5. 고객에게 회사명으로 결제창이 노출되는가?
6. 결제 완료 웹훅 또는 주문 조회 API가 있는가?
7. 결제 완료 데이터에 담당자 코드 또는 커스텀 메타값이 돌아오는가?
```

### 권장 구조

```text
고객
-> 개인 카톡 상담에서 주문 의사 확인
-> 결제 방식 선택: 회사 공식 계좌 입금 또는 회사 카드결제 링크
-> 영업자용 화면에서 주문/결제 링크 생성
-> 고객이 링크에서 주문 정보 직접 입력
-> 회사 결제 확인
-> 회사 배송
-> 영업자 실적 반영
```

### 역할 분리

| 역할 | 담당 |
|---|---|
| 고객 상담 | 영업사원 |
| 고객정보 수집 | 영업사원 |
| 주문 접수 보고 | 영업사원 -> 회사 |
| 결제 수납 | 고객 -> 회사 |
| 카드결제 링크 생성 | 회사 시스템 또는 회사 승인 결제 시스템 |
| 배송/환불/교환 | 회사 |
| 사후 사용 상담 | 영업사원 + 공식 카카오채널 |

### 개인 영업자 입금 구조를 피해야 하는 이유

| 리스크 | 설명 |
|---|---|
| 고객 신뢰 | 개인 계좌보다 회사 공식 계좌가 신뢰도가 높음 |
| 정산 분쟁 | 입금 누락, 금액 오류, 담당자 실적 분쟁 가능 |
| 환불/교환 | 회사가 처리해야 하는 문제를 개인이 떠안게 됨 |
| 세무/증빙 | 개인 입금은 회사 매출/정산 관리가 어려움 |
| 고객정보 | 배송정보와 상담정보가 흩어질 수 있음 |

### 주문 접수 필수 항목

주문 접수는 Google Form, Naver Form, 카카오채널 챗봇, HADa 접수 모듈 등으로 받을 수 있다.

| 항목 | 목적 |
|---|---|
| 입금자명 | 입금 확인 |
| 결제 방식 | 계좌이체 또는 카드결제 링크 요청 구분 |
| 연락처 | 배송/상담 연락 |
| 배송받는 분 | 수령자 확인 |
| 배송 주소 | 배송 처리 |
| 주문 제품 | 제품명 확인 |
| 수량 | 주문 수량 확인 |
| 입금 금액 | 금액 확인 |
| 담당 영업자명/코드 | 실적 반영 |
| 개인정보 수집 동의 | 배송/상담 처리 동의 |
| 카카오채널 안내 동의 | 구매 후 관리/이벤트 안내 |

### 카드결제 처리 방식

카드결제는 영업사원이 개인적으로 받지 않고, 회사가 승인한 시스템에서 공식 결제 링크를 생성해 고객에게 안내하는 구조가 가장 안전하다.  
최종 형태는 본사 직원이 수동으로 링크를 보내는 방식이 아니라, 영업사원이 현장에서 버튼으로 링크를 만들고 카톡에 붙여넣는 방식이다.

```text
영업사원 개인 카톡 상담
-> 고객이 카드결제 요청
-> 영업사원이 영업자용 화면에서 제품/수량/담당자 코드 입력
-> 회사 명의 공식 카드결제 링크 자동 생성
-> 영업사원이 카톡에 결제 링크 붙여넣기
-> 고객이 링크에서 개인정보/배송정보 입력
-> 고객 카드결제 완료
-> 회사 배송
-> 담당자 실적 반영
```

#### 카드결제 링크 후보

| 방식 | 설명 | 비고 |
|---|---|---|
| 폐쇄몰 내부 결제 링크 | 회원/주문 구조와 가장 잘 맞음 | 외부 고객 결제 가능 여부 확인 필요 |
| PG 결제 링크 | 토스페이먼츠, 이니시스, 나이스페이 등 | 회사 명의 PG 계약 필요 |
| 페이앱/블로그페이류 링크결제 | 빠르게 시작 가능 | 회사 명의 계정 권장 |
| 카카오페이 비즈니스 결제 | 카카오 흐름과 연결 가능 | 회사 계정/정책 확인 필요 |

#### 카드결제 운영 원칙

- 카드결제 링크는 회사 명의 결제 시스템에서 생성한다.
- 영업사원은 링크 생성 권한만 갖고, 결제대금은 직접 수납하지 않는다.
- 영업사원 개인 결제 링크나 개인 카드단말기 사용은 원칙적으로 피한다.
- 결제 링크에는 제품명, 수량, 금액, 담당자 코드가 포함되어야 한다.
- 고객 개인정보와 배송정보는 고객이 링크에서 직접 입력하는 구조를 우선한다.
- 환불/교환/배송 문의는 회사 기준으로 처리한다.

### 고객 안내 문구 예시

```text
고객님, 결제는 안전하게 회사 공식 계좌로 진행됩니다.
입금 확인 후 본사에서 주문 접수와 배송을 진행하고,
저는 담당자로서 사용법과 사후 상담을 계속 도와드릴게요.

입금 후 아래 정보를 보내주세요.

입금자명:
주문 제품:
수량:
배송 주소:
연락처:
담당자명:
```

### 카드결제 요청 고객 안내 문구

```text
고객님, 카드결제를 원하시면
회사 공식 카드결제 링크로 안전하게 결제하실 수 있습니다.

아래 링크에서 주문 제품과 수량을 확인하시고
성함, 연락처, 배송주소를 입력하신 뒤 결제하시면 됩니다.

결제와 배송은 회사에서 처리하고,
저는 담당자로서 사용법과 사후 상담을 도와드릴게요.
```

### 결제 방식 선택 안내 문구

```text
결제는 회사 공식 방식으로 진행됩니다.

1. 계좌이체를 원하시면 회사 공식 계좌를 안내드립니다.
2. 카드결제를 원하시면 회사 공식 카드결제 링크를 보내드립니다.

링크에서 주문 제품, 수량, 배송정보를 직접 입력하시면
회사에서 결제 확인 후 배송을 진행합니다.
```

## 5. 담당자 코드 운영

폐쇄몰/네트워크마케팅 구조에서는 영업자별 실적 반영이 중요하다.  
고객이 회사 계좌로 입금하더라도 담당자 코드가 있으면 실적을 반영할 수 있다.

### 담당자 코드 방식

| 방식 | 예시 | 장점 | 주의 |
|---|---|---|---|
| 담당자명 입력 | 홍길동 | 이해하기 쉬움 | 동명이인 가능 |
| 담당자 코드 | HONG01 | 정산 명확 | 코드 안내 필요 |
| 주문폼 선택 | 드롭다운에서 담당자 선택 | 오류 감소 | 담당자 목록 관리 필요 |
| 카카오채널 챗봇 입력 | 담당자 코드를 입력해주세요 | 자동화 가능 | 챗봇 설계 필요 |

### 추천 방식

초기에는 **담당자명 + 담당자 코드**를 함께 받는다.

```text
담당자명: 홍길동
담당자 코드: HONG01
```

나중에 HADa 또는 주문 접수 시스템에서 담당자 코드 자동 검증을 붙인다.

## 6. 공식 카카오채널과 챗봇 역할

공식 카카오채널은 신규 고객의 첫 상담 창구라기보다, 구매고객과 관심고객을 회사 자산으로 전환하는 CRM 채널로 운영한다.

### 구매 후 채널 추가 유도

```text
제품 사용법과 이벤트 안내를 편하게 받아보실 수 있도록
리버스12 세노이드 공식 카카오채널을 추가해 주세요.

채널에서는
제품별 사용 순서,
자주 묻는 질문,
피부 타입별 관리 팁,
재구매 혜택 안내를 받아보실 수 있습니다.
```

### 챗봇 기본 메뉴

| 메뉴 | 내용 |
|---|---|
| 제품 3종 차이 | 미스트/에센스/크림 비교 |
| 사용 순서 | 아침/저녁 루틴 안내 |
| 자주 묻는 질문 | 민감 피부, 사용 주기, 끈적임, 보관법 |
| 구매/배송 문의 | 입금 확인, 배송 일정, 접수 상태 |
| 담당자 연결 | 담당 영업자 또는 본사 상담 연결 |
| 후기 남기기 | 사용 후기 수집 |
| 재구매 안내 | 사용 주기와 혜택 안내 |
| 이벤트 안내 | 체험단, 프로모션, 공지 |

### 챗봇 설계 원칙

- 고객이 가장 자주 묻는 질문부터 배치한다.
- 치료/재생/제거 등 과장 표현은 챗봇 답변에서도 금지한다.
- 제품 사용법, 배송, 재구매 안내처럼 반복 문의를 자동화한다.
- 민감한 피부, 부작용, 환불, 클레임은 사람 상담으로 넘긴다.
- 담당자 코드와 구매 이력을 연결할 수 있으면 재구매 관리가 쉬워진다.

## 8. 교육 과정 반영

### 기초교육

- 브랜드페이지는 "회사 공식 신뢰 확인 페이지"로 소개한다.
- 결제/입금/주문 접수는 자세히 다루지 않고 전체 흐름만 보여준다.
- 카카오톡 상담문에는 "회사 공식 계좌/공식 안내" 표현을 넣는다.

### 중급교육

- 1회차 카카오톡 문구에 회사 공식 결제/입금 안내를 포함한다.
- 카드결제 요청 고객에게는 "회사 공식 카드결제 링크에서 직접 입력/결제"로 안내하는 문구를 포함한다.
- 영업직원에게는 결제 링크를 직접 발송하는 훈련보다, 링크 문구를 복사해 카톡창에 붙여넣는 훈련을 우선한다.
- 2회차 인스타 CTA는 브랜드페이지 또는 블로그 확인으로 연결한다.
- 3회차 블로그 마지막 CTA는 개인 카톡 상담과 브랜드페이지 확인을 연결한다.
- 4회차 7일 캠페인에는 구매 후 공식 카카오채널 추가 유도까지 포함한다.
